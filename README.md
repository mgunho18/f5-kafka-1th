## 과제 1. 현재 비즈니스 구조 분석

### 과제 1-1: 현재 구조 도식화

현재 사내에서 가장 복잡하다고 느끼는 비즈니스 흐름을 하나 골라, 누가 어떤 순서로 무엇을 요청하고 어떤 시스템이 어떤 데이터를 처리하는지 보이도록 정리해 주세요.

- 등장하는 사용자/시스템/외부 연동이 무엇인지
- 요청과 데이터가 어떤 순서로 흐르는지
- 어디서 병목, 대기, 실패 가능성이 생기는지
- 왜 이 흐름이 복잡하다고 판단했는지

> 실제 회사 사례를 쓰기 어렵다면 공개 가능한 유사 도메인으로 바꿔서 작성해도 됩니다.
> 중요한 것은 정답이 아니라 판단 근거입니다.

#### 필수 포함 항목

1. 핵심 시나리오 1개
2. 전체 흐름도 1장
3. 병목 또는 장애 포인트 최소 1개
4. 현재 구조의 한계 최소 1개

#### 권장 주제 예시

- 주문 -> 결제 -> 후속 처리
- 회원 가입 -> 권한 부여 -> 알림 발송
- 정산/배치 처리 -> 결과 반영
- 재고/예약/좌석 같은 동시성 이슈가 있는 흐름

위 예시는 참고용입니다. 실제 회사 흐름을 써도 되고, 공개 가능한 유사 도메인으로 바꿔도 됩니다.

### 과제 1-2: EDA/Kafka 적용 검토

해당 비즈니스에 Kafka를 적용해 볼 수 있는지 고민해 봅니다.
적용이 가능하다면 어떤 식으로 진행할 것인지, 불필요하다면 왜 불필요한지 작성해 봅니다.

> 과제의 핵심은 무조건 Kafka를 도입하라고 말하는 것이 아닙니다.
> 이벤트로 분리하는 편이 나은지, 아니면 현재처럼 처리하는 편이 더 나은지를 스스로 판단해 주세요.

#### EDA/Kafka를 적용해볼 수 있다면

- 어떤 이벤트를 정의할 것인지
- 누가 producer이고 누가 consumer인지
- 왜 동기 호출보다 나은지: 결합도, 확장성, 후속 처리 분리 등
- 운영 시 주의할 점은 무엇인지: 재처리, 멱등성, 순서 보장, 장애 복구 등

#### 적용이 어렵거나 불필요하다면

- 왜 지금 구조가 더 현실적인지
- 어느 지점에서 정합성/순서/운영 부담이 커지는지
- Kafka를 쓰더라도 오히려 복잡도만 늘어나는 이유가 무엇인지

> 중요한 것은 "Kafka가 멋져 보여서 쓴다"가 아니라, 어디까지 이벤트로 풀 수 있고 어디서부터 비용이 커지는지 설명하는 것입니다.

> 현재 상황에서 판단하기 어렵더라도 괜찮습니다. 그것을 배우기 위한 자리입니다.
> 판단하기 어려웠던 이유를 적어 보세요. 수업을 진행하실 때 배워가실 수 있습니다.

## 과제 2. Event Driven Architecture 핵심 개념 정리

Kafka는 EDA의 한 컴포넌트입니다.

Kafka 자체만 보지 말고, Event Driven Architecture라는 구조적 개념을 중심으로 조사해 주세요.

1. Event Driven Architecture란 무엇인가?
2. 어떤 상황에서 특히 유리한가?
3. 대표적인 단점이나 운영 비용은 무엇인가?
4. Kafka는 이 구조 안에서 어떤 역할을 하는가?
5. 내가 고른 비즈니스 흐름에서는 Kafka가 왜 필요하거나, 왜 아직 필요하지 않은가?
